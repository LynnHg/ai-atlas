## 一、核心哲学：从简单开始

文章反复强调的第一原则：

> **找到最简单的可行方案，只在确实能改善结果时才增加复杂度。**

最成功的实现不是用复杂框架，而是用**简单、可组合的模式**构建。大多数场景下，优化单次 LLM 调用 + 检索 + 上下文示例就够了。

---

## 二、Workflow vs Agent 的关键区分

| 类型 | 定义 | 控制权 |
|------|------|--------|
| **Workflow** | LLM 和工具通过**预定义的代码路径**编排 | 开发者控制流程 |
| **Agent** | LLM **动态决定**自己的执行过程和工具使用 | 模型控制流程 |

选择建议：
- **Workflow**：任务明确、需要可预测性和一致性
- **Agent**：需要灵活性和模型驱动的决策
- **大多数场景**：优化单次 LLM 调用 + 检索 + 上下文示例就够了

---

## 三、基础构建块：增强型 LLM

所有 Agent 系统的基础单元 = LLM + 检索 + 工具 + 记忆。

当前模型已经能主动使用这些能力——自行生成搜索查询、选择工具、决定保留哪些信息。MCP 协议提供了与第三方工具生态集成的标准方式。

## 四、五大工作流模式

### 4.1 Prompt Chaining（提示链）

```
任务 → LLM 1 → [检查点] → LLM 2 → [检查点] → LLM 3 → 结果
```

- 将任务分解为**固定的顺序步骤**，每步之间可加程序化检查（gate）
- 用延迟换准确度，让每次调用只做一件简单的事
- 适用：生成营销文案再翻译、先写大纲检查后再写正文

### 4.2 Routing（路由）

```
输入 → 分类 → 路由到专门的处理流程 A / B / C
```

- 对输入分类，导向各自独立优化的 Prompt 和工具
- 关注点分离，避免针对一类输入的优化影响其他类型的表现
- 适用：客服分流（退款/技术支持/一般问题）、简单问题用小模型 + 难问题用大模型

### 4.3 Parallelization（并行化）

两种变体：

| 变体 | 说明 | 示例 |
|------|------|------|
| **Sectioning（切片）** | 任务拆成独立子任务并行执行 | 一个 LLM 处理查询，另一个审核内容安全 |
| **Voting（投票）** | 同一任务跑多次，取多样化输出 | 多个 Prompt 分别审查代码漏洞 |

适用：LLM 在每个关注点由独立调用处理时表现更好。

### 4.4 Orchestrator-Workers（编排者-工人）

```
输入 → 编排者 LLM 动态分解任务 → 派发给多个 Worker LLM → 综合结果
```

- 与并行化的关键区别：子任务**不是预定义的**，而是编排者根据输入动态决定
- 适用：修改多个文件的编码任务、多源信息搜索与分析

### 4.5 Evaluator-Optimizer（评估者-优化者）

```
生成 LLM → 输出 → 评估 LLM → 反馈 → 生成 LLM → 改进 → 循环...
```

- 一个 LLM 生成，另一个评估并给出反馈，循环迭代改进
- 适用条件：有明确的评估标准、人类表达反馈后 LLM 输出能明显改善
- 适用：文学翻译精修、多轮搜索分析

## 五、Agent（自主代理）

```
用户指令 → Agent 规划 → 工具调用 → 观察环境反馈 → 调整 → 循环... → 完成/求助
```

核心特征：
- LLM 在循环中根据**环境反馈**（工具结果、代码执行结果）使用工具
- 每一步都从环境获取 **ground truth** 来评估进展
- 可在检查点暂停请求人类反馈，遇到阻塞时求助
- 需要停止条件（最大迭代次数）防止失控

适用场景：开放式问题、无法预测步骤数、需要模型自主决策、在可信环境中扩展任务。

> Agent 的自主性意味着更高成本和错误复合风险。需要在沙箱中大量测试 + 设置适当的护栏。

---

## 六、模式选择决策指南

| 任务特征 | 推荐模式 |
|----------|----------|
| 可以分解为固定步骤 | Prompt Chaining |
| 输入有明确的分类 | Routing |
| 可以独立并行处理 | Parallelization |
| 子任务不可预测 | Orchestrator-Workers |
| 有明确评估标准需要迭代改进 | Evaluator-Optimizer |
| 开放式、需要自主决策 | Agent |
| 以上都不需要 | **优化单次 LLM 调用就够了** |

这些模式**不是非此即彼**，可以自由组合和定制。

## 七、工具设计：ACI 的重要性

文章提出一个核心类比：

> **在 Agent-计算机接口（ACI）上投入的精力，应该等同于人机交互接口（HCI）。**

### 工具设计原则

1. **站在模型的角度思考**——如果你自己看工具描述都要想半天怎么用，模型也一样
2. **好的工具定义要包含**：示例用法、边缘情况、输入格式要求、与其他工具的边界
3. **参数命名要直观**——就像给初级开发者写文档
4. **大量测试模型如何使用工具**——在 Workbench 中跑大量输入，观察错误模式并迭代
5. **防呆设计（Poka-yoke）**——修改参数设计让模型更难犯错

### 工具格式选择

- 给模型足够的 Token 来"思考"，避免写到一半被困住
- 格式尽量接近模型在互联网文本中见过的自然格式
- 避免格式开销（如精确计算行数、在 JSON 中转义代码）

> SWE-bench 案例：团队在**优化工具上花的时间比优化 Prompt 还多**。发现模型用相对路径会出错，改为强制使用绝对路径后，模型表现完美。

## 八、框架使用建议

- 框架简化了起步，但会创建**额外的抽象层**，让 Prompt 和响应变得不透明，难以调试
- 建议：**先从直接调用 LLM API 开始**——很多模式只需几行代码
- 如果使用框架，**必须理解底层代码**——对底层的错误假设是最常见的错误来源

## 九、实战场景

### 客服支持

- 对话式交互 + 工具集成的天然结合
- 可集成客户数据、订单历史、知识库
- 退款、更新工单等操作可程序化处理
- 成功标准清晰，可按成功解决率定价

### 编码 Agent

- 代码方案可通过自动测试验证
- Agent 可用测试结果作为反馈迭代改进
- 问题空间结构化、输出质量可客观衡量
- 人工审查仍然重要——确保方案符合更广的系统需求

---

## 十、核心启示

1. **最简单的方案优先**——大多数场景不需要 Agent，优化单次调用就够
2. **五大工作流模式是可组合的积木**——不是非此即彼，可以混搭
3. **Workflow 和 Agent 是两种不同的架构**——前者开发者控制，后者模型控制
4. **工具设计 >= Prompt 设计**——ACI 的质量直接决定 Agent 能力上限
5. **防呆设计比写更好的 Prompt 更有效**——改工具比改指令更可靠
6. **框架可以帮助起步，但要理解底层**——生产环境建议减少抽象层
7. **每增加一层复杂度都要有数据证明它改善了结果**
