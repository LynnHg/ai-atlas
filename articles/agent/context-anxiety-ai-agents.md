## 一、什么是"上下文焦虑"（Context Anxiety）

Cognition AI 在用 Claude Sonnet 4.5 重建 Devin 时发现了一个意外现象：**模型会对自己的上下文窗口产生焦虑**。具体表现为：

| 表现 | 说明 |
|------|------|
| **走捷径** | 模型认为快到窗口末尾时，会跳过步骤、留下未完成的任务 |
| **低估剩余 token** | 模型对剩余 token 的估算非常精确——但始终偏低 |
| **主动压缩** | 随着感知到的上下文逼近极限，模型会更频繁地写总结、做笔记 |
| **从并行切换为串行** | 早期激进并行调用工具，后期变得谨慎 |

> Sonnet 4.5 是**第一个具有上下文窗口自我感知能力的模型**。

## 二、上下文压缩的必要性与陷阱

**理论侧**：Anthropic 研究团队指出，对于长期任务，上下文压缩（compaction）——即总结并重新初始化上下文窗口——是必不可少的。Claude Code 的实现会将消息历史交给模型压缩，保留架构决策和未解决的 bug，丢弃冗余输出。

**实践侧**：Cognition 发现 Sonnet 4.5 **自发地**将文件系统当作外部记忆，无需提示就会频繁写总结笔记。但问题在于：

| 策略 | 优点 | 缺点 |
|------|------|------|
| 模型自生成总结 | 自发、无需提示 | **不够全面**，会遗漏重要细节，"模型不知道自己不知道什么" |
| 显式压缩策略 | 结构化、信息保留完整 | 需要开发者设计实现 |

**核心教训**：不要依赖模型的自我总结。生产系统需要**显式的上下文压缩策略**，像结构化会议模板一样确保关键信息不丢失。

## 三、并行执行的双刃剑

Sonnet 4.5 擅长**并行工具调用**——同时执行多个 bash 命令、同时读取多个文件，使会话更高效。但并行执行会**更快消耗上下文**，从而触发上下文焦虑。

模型的行为模式：

- **上下文窗口早期**：激进并行，最大化每轮操作数
- **接近感知极限时**：变得保守，切换为串行

Anthropic 的工具设计指导原则直接回应了这个问题：

- 提供**最小可行工具集**
- 功能**不重叠**——如果人类工程师都无法判断该用哪个工具，AI 也无法判断
- 每次工具调用都有上下文成本，并行执行会使成本倍增

## 四、反馈循环与涌现测试行为

Sonnet 4.5 会**自发地编写和执行测试脚本**，甚至检查 React 应用的 HTML 输出来验证行为——这并非被提示的行为。

这与 Anthropic 的"**即时上下文检索**（just-in-time retrieval）"和**渐进式披露**（progressive disclosure）概念完美吻合：不预先加载所有信息，而是让 Agent 通过工具探索环境，每次交互产生的上下文指导下一步决策。

**注意事项**：Cognition 发现这种涌现行为有时会导致**过度创造性的 workaround**。例如两个本地服务端口冲突时，模型不是简单终止冲突进程，而是创建了过于复杂的自定义脚本。需要提供足够的结构和引导，同时允许有益的涌现行为。

## 五、五个实践策略

| 策略 | 做法 | 原理 |
|------|------|------|
| **1. 启用大窗口，限制实际使用** | 启用 **1M** token 上下文，但实际上限设为 **200k** | 模型觉得还有大量空间，不会触发焦虑行为 |
| **2. 构建显式记忆系统** | 结构化保存架构决策、未解决问题、实现细节 | 模型自己的总结不够全面 |
| **3. 渐进式披露** | 用元数据（目录层级、命名规范、时间戳）作为信号 | 提供上下文但不消耗 token |
| **4. 混合检索策略** | 平衡预计算检索（速度）与运行时探索（准确性） | 动态编码环境偏运行时探索，法律/金融偏前置策展 |
| **5. 监控上下文消耗模式** | 追踪模型何时从并行切换为串行 | 这是触发压缩干预的信号 |

## 核心启示

1. **上下文不只是输入数据，而是一种稀缺的认知资源**——模型自己也在主动管理它
2. **模型的自我感知是不准确的**：Sonnet 4.5 对剩余 token 的估算精确但系统性偏低，开发者不能信任模型的自我评估
3. **"启用大窗口 + 限制实际用量"** 是目前最实用的缓解上下文焦虑的 hack
4. **模型自生成的总结不可靠**——生产系统需要开发者设计显式的、结构化的上下文管理方案
5. 从 prompt engineering 到 **context engineering** 的范式转换已经发生，构建更好的上下文管理系统比写更好的提示词更重要
6. 未来方向：将**上下文焦虑转化为上下文感知**，将上下文限制转化为架构优势
