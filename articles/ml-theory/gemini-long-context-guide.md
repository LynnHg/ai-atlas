## 一、上下文窗口基础

上下文窗口类似于模型的"短期记忆"——你传入多少信息，模型就能处理多少。过去几年的演进：

| 时期 | 模型上下文窗口 | 代表模型 |
|------|---------------|---------|
| 早期 | **8,000** tokens | GPT-3 等 |
| 中期 | **32K–128K** tokens | GPT-4 等 |
| 现在 | **100 万–200 万** tokens | Gemini 1.5 Pro |

**100 万 tokens 实际等于**：

- **50,000 行代码**（每行 80 字符）
- 过去 **5 年**的所有短信
- **8 部**平均长度英语小说
- **200+ 集**播客转写内容

## 二、传统"短上下文"应对策略 vs 长上下文新范式

过去为了应对上下文窗口小的限制，开发者不得不使用以下策略：

| 传统策略 | 做法 | 长上下文时代的变化 |
|----------|------|-------------------|
| 滑动窗口 | 新文本进入时删除旧消息 | 不再需要，直接放入全部内容 |
| 摘要替换 | 将历史内容压缩为摘要 | 可保留完整上下文 |
| RAG + 向量搜索 | 数据移出上下文到向量数据库 | 仍有价值，但不再是唯一选项 |
| 确定性/生成式过滤 | 移除部分文本节省 tokens | 可先尝试全量输入 |

**核心转变**：现在的默认起始操作是**把所有 tokens 直接放入上下文窗口**，而不是先做 RAG。

## 三、长上下文的关键能力——上下文学习（ICL）

Gemini 模型在长上下文窗口下展现出极强的**上下文学习能力**：

- Gemini 1.5 Pro/Flash 仅凭上下文中提供的教学材料（500 页语法 + 字典 + 400 个平行句子），就能学会将英语翻译为 **Kalamang**（一种仅约 200 人使用的巴布亚语言），翻译质量与人类学习者相当
- **检索准确率 > 99%**（大海捞针测试）

## 四、多模态长上下文应用场景

### 长文本场景

| 场景 | 说明 |
|------|------|
| 大型文本摘要 | 无需滑动窗口，一次性处理全部内容 |
| 问答系统 | 无需 RAG 即可直接基于全量上下文回答 |
| Agent 工作流 | 文本是 Agent 保存状态的基础，长上下文让 Agent 更可靠 |
| **多样本上下文学习** | 从几个 shot 扩展到成百上千个 shot，性能接近微调模型 |

**多样本上下文学习（Many-shot ICL）** 是长上下文最独特的能力之一——向模型提供数百、数千甚至数十万个示例，可以产生**接近微调模型**的性能，为生产级场景提供了新的替代方案。

### 长视频场景

- 视频问答、字幕生成
- 视频内存（如 Google Project Astra）
- 视频推荐系统（用多模态理解丰富元数据）
- 视频自定义（移除与观看者无关的视频部分）
- 视频内容审核
- 实时视频处理

### 长音频场景

Gemini 是首批能够**原生理解音频**的多模态大语言模型，无需"语音转文字 + 文本处理"的多模型串联。

| 指标 | Gemini 1.5 Flash | Gemini 1.5 Pro |
|------|------------------|----------------|
| 隐藏音频检测率 | **98.7%** | **100%** |
| 单请求可接受时长 | **9.5 小时** | **19 小时**（200 万 tokens） |
| 字词错误率 (WER) | — | **~5.5%**（低于专用 STT 模型） |

核心优势：原生多模态理解，减少延迟和信息丢失。

## 五、上下文缓存——长上下文的成本优化

**上下文缓存（Context Caching）** 是长上下文的主要优化手段：

- 将用户上传的文件缓存起来，按**小时**付费存储
- 后续请求的输入/输出费用**低于标准价格**
- 特别适合"与数据聊天"场景：用户上传 PDF/视频/文档后多轮对话

## 六、长上下文的局限

- 单根"针"检索准确率 ~**99%**，但**多根"针"**的准确率会下降
- 每次查询都需支付输入 token 费用——检索 100 条信息可能需要 100 个请求
- 使用**上下文缓存**可显著降低重复查询的成本

## 核心启示

1. **"先放进去再说"**：长上下文时代的默认策略是先把所有信息放入上下文窗口，而不是一开始就用 RAG
2. **Many-shot ICL 是微调的替代方案**：提供大量示例可以达到接近微调的效果，且无需训练
3. **多模态原生理解**：Gemini 处理视频/音频不需要先转文字，减少了信息损失和延迟
4. **上下文缓存是必备优化**：对于高输入 token 的工作负载，缓存可大幅降低成本
5. **RAG 没有死**：长上下文不等于 RAG 过时——多针检索的准确率下降意味着 RAG 在精确检索场景仍有价值
6. **成本与准确率的权衡**：99% 的单查询准确率 × 大量查询 = 需要用缓存策略控制总成本
