> 作者：Qishuo Hua, Lyumanshan Ye, Dayuan Fu, Yang Xiao 等（上海交通大学 / SII / GAIR）

## 核心论点：上下文工程 = 熵缩减过程

论文的核心观点是：**上下文工程本质上是一个熵缩减（entropy reduction）过程**——将高熵的人类意图和环境信号转化为低熵的、机器可理解的表示。机器越智能，需要人类做的"预处理"就越少，人机交互成本就越低。

上下文工程不是 LLM 时代的新发明，而是一个跨越 **20+ 年** 的演进学科——从 1990s 的普适计算（Ubiquitous Computing）和上下文感知系统（Context-Aware Systems），到今天基于智能 Agent 的交互范式。

## 四阶段演化模型

论文提出了一个基于机器智能水平的四阶段演化模型：

| 阶段 | 时间 | 机器智能水平 | 上下文角色 | 人机关系 |
|------|------|-------------|-----------|---------|
| **CE 1.0** | 1990s-2020 | 原始计算 | Context as Translation（翻译） | 人适应机器，设计者充当"意图翻译器" |
| **CE 2.0** | 2020-至今 | Agent 智能 | Context as Instruction（指令） | 人机协作，机器能理解自然语言和模糊信息 |
| **CE 3.0** | 未来 | 人类级智能 | Context as Scenario（场景） | 自然协同，AI 能感知社交线索和情绪 |
| **CE 4.0** | 推测 | 超人智能 | Context as World（世界） | 机器引导人类，主动构建上下文 |

**关键演进特征（1.0 → 2.0）：**

| 维度 | CE 1.0 | CE 2.0 |
|------|--------|--------|
| 上下文模态 | 位置、身份、活动、时间 | Token 序列、检索文档、工具 API、用户历史 |
| 核心机制 | 传感器融合、规则触发 | Prompting、RAG、CoT、Memory Agent |
| 上下文容忍度 | 低（结构化输入） | 高（自然语言、图像、视频） |
| 理解方式 | 被动感知 + 条件-动作规则 | 主动理解 + 目标协作 |

## 三大设计维度

### 1. 上下文收集与存储

两个基本设计原则：

- **最小充分性原则（Minimal Sufficiency）**：只收集任务所需的信息，价值在于充分而非量大
- **语义连续性原则（Semantic Continuity）**：保持意义的连续而非数据的连续

存储策略从 1.0 的单设备本地存储演进为 2.0 的分层架构：
- 高频/短期数据 → 快速缓存（内存/边缘节点）
- 中期数据（活动记录、偏好）→ 本地嵌入式数据库（SQLite、LevelDB、RocksDB）
- 长期持久化 → 云存储/远程数据库

**Code Agent 的实践**：Claude Code 维护结构化笔记，定期将关键信息从上下文窗口写入外部记忆，需要时检索——一种轻量但持久的记忆形式。

### 2. 上下文管理

#### 文本上下文处理策略

| 策略 | 优势 | 局限 |
|------|------|------|
| 时间戳标记 | 简单，保留时序 | 无语义结构，线性增长不可扩展 |
| 功能/语义标签 | 便于检索和管理 | 较为刚性，限制灵活推理 |
| QA 对压缩 | 检索效率高 | 破坏原始思路流，不适合需要全面理解的任务 |
| 层次化笔记 | 结构清晰 | 不捕获逻辑关系（因果、证据-结论） |

#### 多模态融合方式

- **共享向量空间映射**：各模态独立编码后投射到统一嵌入空间
- **联合自注意力（Self-Attention）**：文本和视觉 token 在每层相互关注，实现细粒度跨模态对齐
- **交叉注意力（Cross-Attention）**：一种模态作为 query，另一种作为 key/value

#### 分层记忆架构

类比操作系统：LLM 如 CPU，上下文窗口如 RAM——快速但容量有限。

- **短期记忆**：高时间相关性的上下文子集
- **长期记忆**：高重要性且时间相关性已降低的抽象/压缩子集
- **记忆转移**：短期 → 长期的巩固过程，受重复频率、情感重要性和知识关联性驱动

#### 上下文隔离

- **子 Agent 模式**：每个子 Agent 拥有独立的上下文窗口、系统提示和工具权限（如 Claude Code 的 subagent 系统）
- **轻量引用**：大量信息存储在外部，上下文窗口中仅保留引用（sandbox 方法、schema-based state objects）

#### 自烘焙（Self-baking）

自烘焙是将原始上下文转化为紧凑知识结构的过程，这是"记忆存储"与"学习"的分界线。

| 方式 | 说明 | 代表系统 |
|------|------|---------|
| 自然语言摘要 | 存储原始上下文 + 定期生成摘要 | Claude Code, Gemini CLI |
| 固定 Schema 提取 | 提取关键事实到预定义格式（实体图、事件记录、任务树） | ChatSchema, CodeRabbit |
| 直接结构化存储 | 直接以结构化格式存储信息 | HMT |
| 渐进向量压缩 | 编码为稠密向量，周期性摘要合并 | H-MEM |

### 3. 上下文使用

#### Agent 间上下文共享模式

| 模式 | 方式 | 代表系统 |
|------|------|---------|
| 嵌入 Prompt | 将前一个 Agent 的上下文直接嵌入下一个的 prompt | AutoGPT, ChatDev |
| 结构化消息 | 按固定 schema 交换消息 | Letta, MemOS |
| 共享内存 | 通过共享空间间接通信（内存块/黑板/图结构） | MemGPT, A-MEM, G-Memory |

#### 上下文选择关键因素

- **语义相关性**：向量检索最近邻
- **逻辑依赖**：基于推理链的依赖图遍历（MEM1）
- **时效与频率**：近期使用和高频访问的优先级更高
- **去重**：主动检测、合并、移除冗余条目
- **用户偏好反馈**：追踪用户交互模式，动态调整权重

> 经验发现：AI 编码性能在上下文窗口超过 **~50% 填充度**时开始下降——上下文质量 > 上下文数量。

#### 主动用户需求推理

- 分析对话历史和个人数据，学习偏好模式
- 从问题序列推断隐藏目标（如问 Python 装饰器后问性能调优 → 推断目标是提升软件设计效率）
- 检测用户可能卡住的信号，主动提供工具或建议

## 工程实践要点

| 实践领域 | 关键洞察 |
|----------|---------|
| **KV 缓存** | 保持前缀 prompt 稳定（时间戳变化会使缓存失效）；仅追加确定性更新；手动插入缓存检查点；预测性预加载 |
| **工具设计** | 描述必须精确清晰；工具集不宜过大（DeepSeek-v3 超过 **30** 个工具性能下降，超过 **100** 个几乎必定失败） |
| **上下文内容** | 保留错误而非隐藏（有助于纠错学习）；引入结构化微变异（不同序列化模板、措辞、顺序）打破重复模式 |
| **多 Agent** | 主 Agent 必须分配清晰的目标/输出/工具指南/边界；搜索策略从广到窄；扩展思维模式提升准确率 |
| **Todo 技巧** | 维护 todo.md 跟踪子目标；长任务中更新 todo 时用自然语言重述目标，保持在近期上下文中 |

## 应用案例

### CLI Agent（Gemini CLI）

- **GEMINI.md** 层次化文件系统：用户主目录 → 项目根 → 子目录，支持继承和隔离
- 启动时加载静态上下文（系统 prompt、项目环境、GEMINI.md 文件）
- 交互中增量累积动态上下文
- 文件系统本身作为轻量数据库，通过 AI 生成摘要压缩上下文

### Deep Research（通义 DeepResearch）

- 搜索 → 提取关键信息 → 生成子问题 → 整合多源证据
- 周期性调用摘要模型压缩累积历史为"上下文快照"
- 后续推理基于快照而非完整原始历史，突破上下文限制

### 脑机接口（BCI）

- 直接捕获注意力水平、情绪状态、认知负荷等神经信号
- 将上下文收集从外部环境扩展到用户内在认知状态
- 当前技术仍面临噪声和不稳定性挑战

## 未来挑战

1. **上下文收集仍然低效**：过度依赖显式用户输入，BCI 等技术有望突破
2. **大规模上下文存储与管理**：规模增长后检索和选择困难
3. **模型上下文理解有限**：复杂逻辑、图像关系推理仍是短板
4. **长上下文性能瓶颈**：Transformer O(n²) 复杂度；Mamba 等替代架构在长距离推理上仍有弱点
5. **上下文选择精度不足**：有用信号可能被忽略，噪声和冗余信息持续存在
6. **数字存在（Digital Presence）**：人的数字上下文可以在人离去后继续存在并与世界交互——上下文本身成为一种持久的知识、记忆和身份形式

论文提出的终极愿景是构建**语义操作系统（Semantic OS）**——能够像人类大脑一样随时间成长，支持大规模高效语义存储，具备自主添加、修改和"遗忘"知识的能力。

## 核心启示

- 上下文工程不是 LLM 时代的新发明，而是跨越 **20+ 年**的演进学科
- 核心挑战始终是**弥合人类意图与机器理解之间的鸿沟**
- 从"上下文感知（context-aware）"到"上下文协作（context-cooperative）"是 2.0 时代的关键转变
- 实践中，**上下文质量 > 上下文数量**，窗口超过 ~50% 填充度性能开始下降
- 自烘焙（Self-baking）是区分"记忆存储"和"真正学习"的分界线
- 未来方向是构建语义操作系统，具备类人记忆管理能力
